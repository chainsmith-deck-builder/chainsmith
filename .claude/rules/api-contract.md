# API contract rules

The HTTP API is the contract this service offers to clients. As of May 2026, that contract is published as OpenAPI 3.x via `utoipa` macros, emitted to `openapi.json` in CI, and consumed by client repos to generate typed clients.

## utoipa annotations

Every handler exposed in the public router must have:

- A `#[utoipa::path]` attribute with method, path, and a unique `operation_id`
- All request and response types deriving `ToSchema`
- All path and query parameters documented in `params(...)`
- All possible response status codes listed in `responses(...)`, with a schema for each non-trivial body

If a handler can return one of the project's `AppError` variants, the corresponding HTTP status maps to a documented error response with the standard error shape (see Error shape below).

The OpenAPI spec is generated by a small binary (`src/bin/export_openapi.rs`) that imports the router and calls utoipa's spec emitter. CI runs this binary and commits the result as `openapi.json` (or fails the build if the spec is out of date relative to the code).

## Phase-dependent additive rules

Current phase is set in `/CLAUDE.md`. Read it before making API changes.

### Pre-launch (no real users, current phase)

- Break shapes freely. Renaming fields, removing endpoints, restructuring requests is fine.
- Bump the OpenAPI spec but do not bother with deprecation cycles or version bumps.
- Generated TS clients in sibling repos will need a regen. That is expected.

### Production (after real users have data, future phase)

- Additive only. New fields are fine. Removing or renaming a field is a breaking change.
- Deprecate before deleting. Mark the field deprecated in the schema, ship a release, wait at least one full client release cycle, then delete.
- Status code changes for an existing endpoint are a breaking change.
- New required request fields are a breaking change. Default them and make them optional during the deprecation window.
- Major-version bumps go through a versioned URL prefix (`/v2/...`), not silent shape changes on `/v1/...`.

## Error shape

All error responses use the same JSON body:

```json
{
  "error": {
    "code": "string_enum_value",
    "message": "human-readable explanation",
    "details": {}
  }
}
```

- `code` is a stable string clients can switch on. Do not change a code without a deprecation cycle in production.
- `message` is for humans and may change without warning.
- `details` is optional structured context (e.g. validation failures listing the offending fields).

Error codes are defined as an enum in `error.rs` with a `#[serde(rename_all = "snake_case")]` attribute. Do not invent ad-hoc strings in handlers.

## JSON conventions

- Field names are `camelCase` on the wire. Apply via `#[serde(rename_all = "camelCase")]` on each request and response struct.
- Dates and times are RFC3339 strings (`2026-05-02T14:23:00Z`). Never Unix timestamps in the wire format.
- Optional fields: prefer omitting absent fields over sending `null`, unless `null` carries a meaning distinct from absence. Use `#[serde(skip_serializing_if = "Option::is_none")]`.
- Enums use `#[serde(rename_all = "snake_case")]` unless there is a reason to differ.

## Pagination

Cursor-based, not offset-based. Endpoints returning lists accept `limit` and `cursor` query params. Responses include `nextCursor` (string or null). The cursor is opaque to the client.

## Auth

Routes that require authentication go behind the auth middleware (see `auth/middleware.rs`). Public routes are explicitly registered as such. The default is authenticated.

## Rate limits

Not enforced in pre-launch. In production, document any rate-limit headers and their semantics in the OpenAPI spec when they are added.

## Backwards-compatibility checklist (production phase)

Before merging any change to the public API in production, confirm:

- No field has been removed from any response type
- No field has been renamed in any request or response type
- No new field has been added as required to any request type
- No status code has changed for an existing operation
- No `operation_id` has changed
- The change is reflected in `openapi.json` and the diff against the prior spec is reviewed

## Client distribution

Typed clients are generated by each consumer repo from the committed `openapi.json` in this repo. This service does not publish typed packages of its own.

### Pinning

Each consumer repo pins to a specific commit of this repo via a single file (e.g. `.api-version` containing the SHA). Generation reads the spec from:

```
https://raw.githubusercontent.com/chainsmith-deck-builder/chainsmith/<sha>/openapi.json
```

A scheduled job (Renovate custom manager or a weekly GitHub Action) bumps the SHA to `main` HEAD, regenerates the client, and opens a PR. If the regenerated types break the consumer's typecheck, the PR fails loudly — that is the intended drift signal.

### Recommended toolchains

- **TypeScript** (web client): `openapi-typescript` for types-only generation (`schema.d.ts`) plus `openapi-fetch` as the runtime wrapper. The committed `schema.d.ts` is auditable in PR diffs and keeps builds offline. The stable `ErrorCode` enum surfaces as a string-literal union for exhaustiveness-checked `switch (err.code)`.
- **Swift** (future iOS client): `swift-openapi-generator` against the same pinned SHA.
- **Kotlin** (future Android client): `openapi-generator-cli --generator-name kotlin` against the same pinned SHA.

Each consumer owns its `.api-version` and its generation script. The backend's only obligation is to keep `openapi.json` correct and committed, which the existing CI drift check and pre-commit hook already enforce.

### Operation grouping

Handlers carry `tags = [...]` in their `#[utoipa::path]` macros (`Health`, `Catalog`, `Validation`, `Decks`). Generators use these to group operations into namespaced clients (e.g. `client.decks.create(...)` in TS, separate protocols in Swift). Tag descriptions live on the `ApiDoc` derive in `src/api/mod.rs`.
